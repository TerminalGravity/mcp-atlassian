## Defects

- **B1 duplicate create** — session `0c238ee8`: identical back-to-back create_issue → DS-12909 created twice. No idempotency guard. OPEN.
- **B2 write succeeded, response errored** — 2026-05-12 dogfood: 3/5 assignee writes succeeded though the tool returned token-limit errors; agent re-ran them. Response failure must never mask write success. PARTIAL.
- **B3 Markdown↔Wiki mangling** — 20 edit_comment calls are render repairs; every comment post triggers a verify-via-get_issue. format='auto' preflight warns pre-send; no rendered preview in response. PARTIAL.
- **B4 assignee no-op** — FIXED (jira_assign_issue, 67 adoptions).
- **B5 hard errors are rare** — 4 files with permission errors corpus-wide; zero invalid-transition/accountId/rate-limit. Friction is round-trips, not failures.

## Round-trip waste

- **C1 same-issue re-fetch** — DS-12400 ×8, DS-12515 ×6, three others ×3 in single sessions. ~30–40% of get_issue calls are redundant.
- **C2 post-comment verification** — up to 1 extra get_issue per 230 comments.
- **C3 transition-ID lookup** — 103 get_transitions exist only to map status names to opaque IDs for 223 transitions. 100% eliminable.
- **C4 search→get_issue fan-out** — 5–10 follow-up fetches per triage search for fields the search omitted.
